# JC3636K718 Controller Notes

This repository does not include the raw vendor bundle `JC3636K718_knob_EN/` because the local package is about `637 MB` and mostly consists of binaries, flashing tools, PDFs, logs, and demo projects.

This document captures the parts that matter for firmware work on the round La Marzocco controller so the project can be continued from a remote or cloud environment without redistributing the original vendor package.

## Project use

For this project, the board is intended to run a custom firmware for:

- round `360x360` controller UI
- rotary encoder rotation for value changes
- touch input for field selection, presets, and setup
- Wi-Fi setup and local config portal
- La Marzocco cloud login and machine selection
- later optional BLE integration

## Verified hardware facts

The following data was extracted from the local vendor examples and board config headers.

### MCU / board family

- The bundle includes an `ESP32-S3R8` chip manual, so this board is very likely based on an `ESP32-S3` variant.
- The example projects are provided for both `Arduino` and `ESP-IDF`.

Status:
- Treat `ESP32-S3` as confirmed enough for firmware planning.
- Verify exact flash / PSRAM SKU on real hardware before final memory budgeting.

### Display

- Controller display resolution: `360 x 360`
- Display controller: `ST77916`
- Color depth in example config: `16 bpp`
- Bus style in the IDF example: QSPI LCD

Relevant example facts:

- `lv_obj_set_size(..., 360, 360)` appears in the vendor LVGL demos.
- LCD config uses:
  - `LCD_H_RES 360`
  - `LCD_V_RES 360`
  - `LCD_BIT_PER_PIXEL 16`

### Touch

- Touch controller in the IDF example: `CST816S`
- Touch bus: `I2C_NUM_0`
- Touch pins:
  - `GPIO 9` SDA
  - `GPIO 10` SCL
  - `GPIO 7` INT
  - `GPIO 8` RST

This means swipe support is realistic in the final firmware if we want it.

### Rotary encoder and optional push button

From the vendor demos:

- rotary encoder A: `GPIO 2`
- rotary encoder B: `GPIO 1`
- encoder push button: `GPIO 0`

The vendor example code also shows explicit support for press / release / long press semantics:

- LVGL event handling for `LV_EVENT_PRESSED`, `LV_EVENT_RELEASED`, `LV_EVENT_LONG_PRESSED`
- example button timing:
  - `short_press_time = 200`
  - `long_press_time = 1000`

Important project note:

- the real installed La Marzocco controller variant currently available for this project does **not** expose a usable encoder press action
- current firmware should therefore assume **rotation only** on the encoder and use touch for selection and preset actions
- `GPIO0` remains relevant as a vendor-board clue, but should not be treated as a required runtime input on this hardware

### Backlight

- LCD backlight pin: `GPIO 21`

This should be driven by PWM so brightness can be controlled in settings.

## Important caution

There is one pin conflict worth verifying on the actual hardware:

- The Arduino demo uses `BTN_PIN = 0`
- The IDF `pinconfig.h` also defines `PIN_RGB_DATA = GPIO_NUM_0`

That may indicate one of the following:

- board revision differences
- example-code inconsistency
- multiplexed use that we should avoid

Before finalizing any `GPIO0` assumptions, we should test the real board and confirm:

- whether the vendor demo's encoder push switch is really on `GPIO0`
- whether there is also an RGB LED on that line
- whether `GPIO0` has any boot-mode implications on this board revision

## Recommended firmware stack

Recommended first implementation:

- `ESP-IDF`
- `LVGL`
- custom app state machine
- local persistence via `NVS`
- Wi-Fi setup via captive portal or local AP
- local web config page for:
  - Wi-Fi
  - La Marzocco cloud login
  - machine selection
  - later BLE token / local pairing data

Why this is the right first cut:

- the UI can stay simple: tap to select, rotate to change
- it matches the installed hardware better than a press-driven flow
- Wi-Fi + cloud path already works conceptually in the browser simulator
- it avoids blocking firmware work on unresolved BLE reverse engineering

## Recommended app structure

Suggested firmware modules:

- `app_main`
- `ui/`
  - round screens
  - presets
  - settings flow
- `input/`
  - encoder
  - touch gestures
- `services/`
  - Wi-Fi
  - captive portal
  - storage
  - La Marzocco cloud client
  - optional BLE client later

## Suggested setup flow

For the real device, credentials should not be typed character-by-character on the knob UI.

Recommended first-boot flow:

1. Device starts a local AP, for example `LM-CTRL-XXXX`
2. Round screen shows:
   - SSID
   - setup URL or IP (`192.168.4.1`)
   - QR code to open the setup page
3. User enters:
   - home Wi-Fi
   - La Marzocco account email
   - La Marzocco password
4. Device fetches cloud machines
5. User selects the machine
6. Device stores the configuration locally

## What is still missing

These points still need real-device confirmation:

- exact board SKU / memory size
- whether touch is reliable enough to stay as the primary selection path
- whether the installed controller variant exposes any usable press input at all
- whether `GPIO0` is wired to anything user-accessible on this build
- brightness range and acceptable PWM settings
- whether the board's bundled demos rely on any hidden init steps not yet captured here
